# ETF雷达 - 架构设计

## 一、架构决策与理由

### 数据源选型

经过实际验证（2026-04-07），确定以下数据源方案：

| 数据源 | 用途 | 优势 | 劣势 |
|--------|------|------|------|
| **腾讯财经 qt.gtimg.cn** (主) | 每日份额采集 | 支持批量请求、无明显限流、返回总市值 | 无历史数据 |
| **AKShare fund_etf_hist_em** (辅) | 历史行情回补 | 接口封装好、数据完整 | 底层调东方财富，有频率限制 |
| **东方财富 pingzhongdata** (校准) | 季度规模校准 | 官方数据、含环比 | 粒度为季度 |

**份额计算方式**: `份额(亿份) = 总市值(亿元) / 最新价(元)`

### 为什么不用 Tushare / 天天基金 FundSize

- Tushare Pro 需要积分门槛，不适合开源项目
- 天天基金 FundSize API 对场内ETF返回404，仅支持场外基金
- 东方财富 push2 接口短时间频繁请求会被限流（RemoteDisconnected）

### 为什么选 SQLite 而不是 PostgreSQL

- 本项目定位为**本地部署工具**，SQLite 零配置、单文件、备份方便
- ETF数据量小（几十只ETF × 365天/年 ≈ 万级记录），SQLite 完全够用
- 后期如需迁移到 PostgreSQL，SQLAlchemy ORM 层切换成本极低

## 三、数据采集策略

### 增量更新机制

```
启动时:
  1. 检查 etf_share 表最新日期
  2. 如果 < 今天 且今天是交易日:
     → 采集今天的数据 (增量)
  3. 如果表为空:
     → 通过 AKShare 回补历史数据

每日定时 (16:00):
  1. 判断今天是否为交易日
  2. 查询 etf_share 表是否已有今天数据
  3. 如果没有 → 采集并写入
  4. 如果有 → 跳过 (幂等)

手动触发:
  POST /api/v1/collect/trigger
  → 强制采集当天数据 (覆盖写入)
```

### 数据采集流程

```
1. 从 etf_fund 表读取所有 is_active=true 的ETF
2. 拼接腾讯财经批量查询URL (一次请求最多50只)
3. 解析返回数据，提取: 代码、名称、最新价、总市值
4. 计算份额 = 总市值 / 最新价
5. 写入 etf_share 表 (ON CONFLICT 更新)
```

## 四、API 设计 (v1)

所有 API 以 `/api/v1` 为前缀，返回统一 JSON 格式，方便 Web 和小程序共用。

### ETF管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/funds | 获取ETF列表 |
| POST | /api/v1/funds | 添加ETF |
| PUT | /api/v1/funds/{code} | 更新ETF信息 |
| DELETE | /api/v1/funds/{code} | 删除ETF |

### 份额数据

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/shares | 查询份额数据 (支持 code/group/start/end 筛选) |
| GET | /api/v1/shares/latest | 获取所有ETF最新份额 |
| GET | /api/v1/shares/summary | 分组汇总 (如沪深300组合计) |
| GET | /api/v1/shares/trend | 趋势数据 (前端图表用) |

### 采集控制

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/v1/collect/trigger | 手动触发采集 |
| GET | /api/v1/collect/status | 采集状态 (最近采集时间/结果) |

### 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

## 五、数据库设计

### etf_fund (ETF基金表)

| 字段 | 类型 | 说明 |
|------|------|------|
| code | TEXT PK | 基金代码 (如 510300) |
| name | TEXT | 基金名称 |
| market | TEXT | 市场 (sh/sz) |
| index_name | TEXT | 跟踪指数 (如 沪深300) |
| group_tag | TEXT | 分组标签 |
| is_active | BOOLEAN | 是否启用采集 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### etf_share (份额数据表)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| fund_code | TEXT | 基金代码 (FK) |
| trade_date | DATE | 交易日期 |
| price | REAL | 最新价 |
| total_market_cap | REAL | 总市值 (亿元) |
| shares | REAL | 份额 (亿份) |
| change_shares | REAL | 份额变化 (亿份, 与前一日差) |
| source | TEXT | 数据来源 (tencent/akshare/manual) |
| created_at | DATETIME | 采集时间 |

**唯一约束**: `UNIQUE(fund_code, trade_date)` — 保证幂等写入

### collect_log (采集日志表)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| trade_date | DATE | 采集日期 |
| status | TEXT | success/failed/partial |
| fund_count | INTEGER | 采集ETF数量 |
| message | TEXT | 详情/错误信息 |
| created_at | DATETIME | 采集时间 |

## 六、微信小程序接入方案

当前架构已为小程序接入做好准备：

### 已具备的条件

1. **RESTful API + JSON 响应** — 小程序 `wx.request` 可直接调用
2. **API 版本化 (/api/v1/)** — 后续可加 v2 而不影响现有客户端
3. **CORS 已开启** — 跨域无障碍
4. **无状态 API** — 不依赖 session，天然适合多端

### 小程序接入时需要做的

1. **公网部署** — 本地 SQLite 方案不变，加一层 Nginx 反代 + HTTPS
2. **域名备案** — 小程序要求后端域名已备案
3. **认证 (可选)** — 如需多用户，加 JWT token 认证中间件
4. **小程序前端** — 推荐 uni-app (可同时生成H5+小程序)

### 架构不需要改的

- 后端 API 层完全复用
- 数据库不需要改
- 采集逻辑不需要改
- 如果用户量大了，SQLite → PostgreSQL 只需改一行连接字符串

## 七、与类似项目的对比

| 维度 | 本项目 | 常见量化数据项目 |
|------|--------|-----------------|
| 定位 | ETF份额追踪专用工具 | 通用金融数据平台 |
| 数据源 | 腾讯财经(免费/无限流) | Tushare Pro(需积分) |
| 部署 | 本地单机, 零配置 | 通常需要 Docker/云服务 |
| 数据库 | SQLite (单文件) | PostgreSQL/MySQL |
| 前端 | Vue3 SPA | 多为 Jupyter/命令行 |
| 小程序 | API 层已预留 | 通常不考虑 |
| 增量更新 | 有 (幂等写入) | 多为全量刷新 |
